# Deploy-харнесс для live-демо (Tages 29.06 / KTS 01.07)

Цель: на собесе агенты собирают сервис → ты **одной командой** выкатываешь его в публичный
HTTPS-URL на своём домене. Деплой детерминирован и пред-протестирован — стохастическая часть
(агенты) его не трогает.

## Что пред-запечено (делаешь ОДИН раз, заранее)
1. VPS (Ubuntu), Docker установлен.
2. DNS: A-запись `demo.<домен>` → IP VPS (сделать заранее — пусть пропагация пройдёт).
3. Caddy поднят и держит Let's Encrypt-cert на `demo.<домен>` (см. `docker-compose.caddy.yml`).
   Caddy остаётся жить постоянно, между прогонами не трогаем.

Разовый прогон на VPS:
```bash
# на VPS, в /opt/demo, положив рядом Caddyfile + docker-compose.caddy.yml + .env
docker compose -f docker-compose.caddy.yml up -d   # создаёт сеть demonet, поднимает edge
curl -I https://demo.<домен>/   # 502 ожидаем (app ещё нет) — но TLS должен встать
```
Если TLS встал и видишь 502 от Caddy — инфра готова. Дальше живой путь только: build app → up.

## Что строят агенты вживую (контракт приложения)
Приложение в каталоге, который агенты наполнят. Жёсткие требования (контракт деплоя):
- слушает `0.0.0.0:8000`;
- entrypoint модуль `main:app` (FastAPI/ASGI);
- есть `GET /health` → 200;
- есть `requirements.txt` (минимум `fastapi`, `uvicorn[standard]`).
Всё остальное (роуты, LLM-эндпоинт, фронт в `static/`) — свобода агентов.

Dockerfile НЕ пишут агенты — он фиксирован (`./Dockerfile`), копируется поверх при деплое.

## Живой путь на собесе (одна команда)
```bash
./deploy.sh <путь-к-собранному-агентами-приложению>
# rsync → docker compose up -d --build на VPS → healthcheck https://demo.<домен>/health
# на успехе печатает: ✅ LIVE: https://demo.<домен>
```

## Перед собесом — обязательно
- [ ] DNS пропагирован, `https://demo.<домен>/` отдаёт TLS (502 ок).
- [ ] `./deploy.sh ./sample-app` прогнан минимум 2 раза до зелёного healthcheck.
- [ ] `.env` заполнен (см. `.env.example`), ключи на месте.
- [ ] SSH-доступ к VPS без пароля (ключ), `ssh $VPS_HOST` заходит молча.
